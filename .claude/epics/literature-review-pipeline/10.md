# Refactor: query-scoped storage for literature review runs

## Context

Current implementation stores all search results, downloaded PDFs, converted Markdown, notes, and outputs in shared global directories under `data/` and `outputs/`. This causes different keyword-based literature review runs to mix together.

## Description

Refactor the pipeline to create one isolated directory per search query / review run, so each topic has its own raw results, papers, Markdown conversions, notes, and final review output.

## Desired Structure

```text
data/queries/<query-slug>/
├── search-results/
│   ├── raw.json
│   ├── filtered.md
│   └── fetch-status.md
├── papers/
├── papers-md/
├── notes/
└── outputs/
    └── review-overview.md
```

## Acceptance Criteria

- [x] Query slug is derived deterministically from keywords
- [x] search-agent writes all artifacts into `data/queries/<slug>/...`
- [x] literature-reviewer reads/writes from the selected query directory
- [x] synthesis-agent reads notes and writes final review within the selected query directory
- [x] Existing global `data/` paths are no longer the default for new runs
- [x] Architecture and implementation docs are updated
- [x] GitHub issue created and linked to epic `literature-review-pipeline`
